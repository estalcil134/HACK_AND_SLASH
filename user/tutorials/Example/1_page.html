Tutorial: Web Exploitation

 

SQL Injection:

SQL Injection is a malicious code injection that can destroy a site’s internal or external database. It is a very common hacking technique due to the accessibility and efficiency that it can retrieve private information if the database is not secure. The most common SQL injection will happen when hackers input SQL statements into website inputs such as username and password. The injection will then run and process the statement into the database and return information such as passwords or emails. This process will unknowingly run on the database which is a threat to user accounts and compromise the website as a whole.

 

Examples of SQL injection statements:

·      “SELECT * FROM Users WHERE ID = 10 OR 1=1”: This statement will always return true because 1 = 1 is true. It will result in all the user information

·      “SELECT * FROM Users; DROP TABLE Suppliers”: Will return all information from the Users table while deleting the Suppliers table. This is called a batched SQL statement.

·      SELECT * FROM Users WHERE Name =”” OR “”=”” AND Password =”” or “”=””: The statement will always return valid and will return all information from the Users table because OR “”=”” will return True.

 

Now how can you prevent a SQL injection on your own site/database?

·      A site manager can use SQL parameters which are values that are added to a SQL run statement which is controlled. In this case, the parameters are used by a ‘@’ symbol

o   “SELECT * FROM Users WHERE ID = @0”

o   The database will check the parameters to make sure that the value is correct and will not execute as one SQL statement

o   “db.Execute(statement, name, address, city)”

o   PHP prepared statements are another layer of security

 

Cross Site Scripting:

Cross Site Scripting is an injection attack, where malicious scripts made by hackers are inserted into “safe” websites. These attacks occur the most when a hacker uses a web application to send code (browser-side script) to a user on the other end. Websites have flaws that allow these scripting attacks to process through the users end and execute. When the script is sent to the user unknowingly, the browser is not able to process its validity and then will be executed. As a result, the user’s browser believes that the malicious script is trustworthy then will be able to get any of the user’s private information (cookies, login, tokens). Even so, these malicious attacks can even redo the website’s internal code.

 

How to prevent an XSS attack:

·      Turn off HTTP TRACE support on all webservers

·      Use caution into HTML and/or JavaScript before inserting untrusted data

·      JSON serialization

·      More options could be found at https://www.owasp.org/index.php/XSS_(Cross_Site_Scripting)_Prevention_Cheat_Sheet
